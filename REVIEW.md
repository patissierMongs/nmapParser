## Q2) 코드 리뷰 재점검

요청 기준:
1. 개인정보
2. 불필요한 코드
3. 잘못 설계된 코드
4. 저작권 이슈

### 1) 개인정보(PII) 점검 결과
- 하드코딩된 비밀번호/토큰/API Key/개인식별 정보는 확인되지 않음.
- 단, 스캔 결과 파일(`.log/.xml/.csv`)은 대상 자산 정보(IP/서비스 배너)를 담으므로 운영상 민감정보 취급 필요.

### 2) 불필요한 코드 점검 결과
- 기능에 직접 영향 없는 치명적 dead code는 뚜렷하지 않음.
- 다만 `_append_log()`와 `_append_log_batch()`가 rolling-buffer 정리 로직을 일부 중복 보유하므로,
  후속 리팩터링 시 공통 함수화 여지 있음.

### 3) 설계 점검 결과
- 장점
  - `subprocess.Popen(cmd_list)` + `shell=True` 미사용으로 명령 주입 리스크를 낮춘 구조.
  - 검증/파싱/충돌감지를 함수 단위로 분리해 유지보수성 개선됨.
- 보완 포인트
  - 옵션 충돌 규칙(`OPTION_CONFLICT_PREFIXES`)은 현재 TCP 스캔 타입 위주라, 향후 확장형 규칙 모델로 발전 필요.
  - 로그 배치 파이프라인은 도입 완료했으므로, 실제 대용량 출력에서 배치 파라미터 튜닝(주기/배치량) 권장.

### 4) 저작권 이슈 점검 결과
- 저장소에 MIT 라이선스 파일이 존재하며, 명시적 저작권 고지/허가 문구 포함.
- 외부 라이브러리 의존은 표준 라이브러리 중심이며, 별도 서드파티 코드 복붙 흔적은 현재 점검 범위에서 발견되지 않음.
- README의 도구명/프로젝트명 언급은 일반적 참조 수준으로 보이며, 코드 자체의 라이선스 충돌 정황 없음.

## 결론
- 현재 상태는 배포 전 "마감 점검 단계"로 판단.
- 남은 리스크는 구조적 결함보다는 **검증 시나리오 확장과 운영 가이드(민감정보 관리)** 쪽에 가까움.
